================================================================================
                            CHAPTER 1
                          INTRODUCTION
================================================================================

1.1 PROJECT OVERVIEW
--------------------
HelpRadar is a full-stack web application designed to connect community members
during emergencies, lost item situations, blood donation needs, and volunteer
opportunities. The platform enables hyperlocal communication by allowing users
to post requests for help and receive responses from people in their immediate
geographical vicinity.

The application leverages modern web technologies including Next.js 16, React 19,
MongoDB, and TypeScript to deliver a responsive, real-time, and secure platform
for community assistance.

1.2 PROBLEM STATEMENT
---------------------
In urban communities, people often face emergency situations where they need
immediate help from those around them. Current solutions have several limitations:

1. LACK OF HYPERLOCAL FOCUS
   - Social media platforms are too broad and not location-specific
   - Emergency services may not cover all types of community needs
   - Neighborhood groups on WhatsApp/Facebook lack proper categorization

2. BLOOD DONATION CHALLENGES
   - Finding blood donors in urgent situations is time-critical
   - Existing platforms don't prioritize urgency effectively
   - No immediate notification system for potential donors

3. LOST ITEM RECOVERY
   - No dedicated platform for reporting lost items locally
   - Found items have no centralized place to be reported
   - Lack of contact mechanism between finder and owner

4. COMMUNITY DISCONNECTION
   - People willing to help don't know who needs assistance
   - Those in need don't know who can help in their area
   - No trust mechanism for community interactions

1.3 PROPOSED SOLUTION
---------------------
HelpRadar addresses these problems by providing:

1. HYPERLOCAL PLATFORM
   - City and area-based filtering
   - GPS coordinates for precise location
   - Map visualization of nearby requests

2. SMART PRIORITIZATION
   - Urgency-based categorization (Low/Medium/High)
   - AI-like keyword analysis for auto-categorization
   - Priority scoring for feed ranking

3. REAL-TIME NOTIFICATIONS
   - In-app notification bell with unread count
   - Email alerts for urgent requests
   - Instant helper contact notifications

4. SECURE COMMUNICATION
   - Masked contact information
   - Helper messaging system
   - Report and moderation features

1.4 PROJECT OBJECTIVES
----------------------
PRIMARY OBJECTIVES:
- Develop a user-friendly web platform for community help requests
- Implement secure authentication with multiple providers
- Create an efficient notification system for real-time alerts
- Build an admin dashboard for platform moderation

SECONDARY OBJECTIVES:
- Implement smart categorization using keyword analysis
- Develop interactive map visualization
- Create a post lifecycle with auto-expiry
- Enable community self-moderation through reports

2.1 EXISTING SYSTEMS
--------------------

2.1.1 SOCIAL MEDIA PLATFORMS (Facebook, Twitter)
Advantages:
- Large user base
- Easy sharing mechanism
- Mobile applications

Disadvantages:
- Not hyperlocal
- No categorization for help types
- Requests get lost in feed
- No urgency prioritization

2.1.2 NEXTDOOR
Advantages:
- Neighborhood-focused
- Verified addresses
- Local recommendations

Disadvantages:
- Limited to residential areas
- No emergency prioritization
- No blood donation features
- Limited in Indian context

2.1.3 BLOOD DONATION APPS
Advantages:
- Focused on single purpose
- Blood type matching
- Hospital integration

Disadvantages:
- Only for blood donation
- No general help features
- Limited user base

2.2 RESEARCH GAP
----------------
Current solutions either focus on general social networking without hyperlocal
features, or are too specialized for single purposes. HelpRadar bridges this
gap by providing:

- Multi-category support (Help, Lost Items, Blood, Offers)
- Hyperlocal focus with GPS integration
- Smart prioritization without complex AI
- Modern, responsive user interface
- Secure, privacy-focused design

3.1 FUNCTIONAL REQUIREMENTS
---------------------------

FR1: USER AUTHENTICATION

FR4: NOTIFICATIONS
    - FR4.1: Users shall receive in-app notifications

    - FR4.3: Post creators shall be notified when helpers contact them

    - FR5.1: Users shall be able to view post creator contact

4.2 DATABASE DESIGN
-------------------

COLLECTIONS:

1. USERS COLLECTION
   {
     _id: ObjectId,
     name: String,
     email: String (unique),
     password: String (hashed, select: false),
     googleId: String,
     role: String (enum: user, admin),
     trustScore: Number (0-100),
     resetOtp: String,
     resetOtpExpiry: Date,
     subscribedCities: [String],
     subscribedCategories: [String],
     notificationPreferences: {
       email: Boolean,
       bloodAlerts: Boolean,
       helpAlerts: Boolean
     },
     createdAt: Date,
     lastLogin: Date
   }

2. POSTS COLLECTION
   {
     _id: ObjectId,
     title: String (max: 150),
     description: String (max: 2000),
     category: String (enum: Help Needed, Item Lost, Blood Needed, Offer),
     city: String,
     area: String,
     location: {
       type: String (Point),
       coordinates: [Number, Number] (lng, lat)
     },
     contact: {
       name: String,
       phone: String,
       email: String
     },
     urgency: String (enum: Low, Medium, High),
     images: [{
       url: String,
       public_id: String
     }],
     status: String (enum: active, resolved, removed),
     views: Number,
     reported: Number,
     priority: Number,
     createdBy: ObjectId (ref: User),
     createdAt: Date,
     expiresAt: Date (TTL)
   }

3. NOTIFICATIONS COLLECTION
   {
     _id: ObjectId,
     recipient: ObjectId (ref: User),
     recipientEmail: String,
     type: String (enum: help_offered, new_post, post_resolved, system),
     title: String (max: 200),
     message: String (max: 500),
     postId: ObjectId (ref: Post),
     postTitle: String,
     senderId: ObjectId (ref: User),
     senderName: String,
     senderEmail: String,
     isRead: Boolean,
     urgency: String,
     createdAt: Date (TTL: 30 days)
   }

4. REPORTS COLLECTION
   {
     _id: ObjectId,
     postId: ObjectId (ref: Post),
     reason: String (enum: spam, inappropriate, false_info, harassment, other),
     description: String (max: 500),
     reportedBy: ObjectId (ref: User),
     status: String (enum: pending, reviewed, dismissed),
     createdAt: Date
   }

DATABASE INDEXES:
- 2dsphere index on posts.location (geospatial queries)
- Text index on posts.title and posts.description (full-text search)
- Compound index on posts.city, category, urgency (filtering)
- TTL index on posts.expiresAt (auto-cleanup after 7 days)
- TTL index on notifications.createdAt (auto-cleanup after 30 days)
- Index on users.email (authentication)
- Index on notifications.recipientEmail (notification fetch)

4.3 API DESIGN
--------------

REST API ENDPOINTS:

AUTHENTICATION:
  POST   /api/auth/register        - Create new user account
  POST   /api/auth/login           - Authenticate user
  POST   /api/auth/forgot-password - Send OTP to email
  POST   /api/auth/verify-otp      - Verify OTP code
  POST   /api/auth/reset-password  - Set new password
  *      /api/auth/[...nextauth]   - NextAuth.js handlers

POSTS:
  GET    /api/posts                - List posts with filters
  POST   /api/posts                - Create new post
  GET    /api/posts/[id]           - Get single post
  PATCH  /api/posts/[id]           - Update post
  DELETE /api/posts/[id]           - Delete post
  POST   /api/posts/[id]/contact   - Contact post creator
  POST   /api/posts/[id]/report    - Report post

NOTIFICATIONS:
  GET    /api/notifications              - Get user notifications
  PATCH  /api/notifications/[id]         - Mark notification read
  POST   /api/notifications/mark-all-read - Mark all read

STATISTICS:
  GET    /api/stats    - Get aggregated statistics
  POST   /api/suggest  - Get smart suggestions

5.2 KEY FEATURES IMPLEMENTATION
-------------------------------

5.2.1 AUTHENTICATION SYSTEM

The application implements dual authentication:

A) Email/Password Authentication:
   - Registration with validation
   - Password hashing using bcryptjs
   - Login with session creation
   - Password reset via OTP

B) Google OAuth:
   - NextAuth.js Google provider
   - Automatic user creation
   - Account linking for existing users

Code Structure:
- /src/auth.ts - NextAuth configuration
- /src/app/api/auth/* - Auth API routes
- /src/contexts/AuthContext.tsx - Auth state management

5.2.2 POST MANAGEMENT

Posts are managed through a comprehensive system:

A) Creation:
   - Multi-step form wizard
   - Input validation
   - Image upload to Cloudinary
   - Smart priority calculation
   - User ID association

B) Display:
   - Paginated list view
   - Card-based UI
   - Category color coding
   - Urgency indicators
   - View count tracking

C) Lifecycle:
   - Auto-expiry (7 days TTL)
   - Mark as resolved
   - Report and moderation
   - Status tracking

5.2.3 NOTIFICATION SYSTEM

Dual notification mechanism:

A) In-App Notifications:
   - Stored in MongoDB
   - Real-time bell icon updates
   - Unread count badge
   - Dropdown preview
   - Full notifications page

B) Email Notifications:
   - Urgent post alerts
   - Helper contact messages
   - OTP for password reset
   - HTML email templates

5.2.4 SMART PRIORITY ENGINE

Keyword-based analysis without ML:

A) Category Detection:
   - Blood keywords: blood, donor, A+, B+, O+, transfusion
   - Lost keywords: lost, missing, stolen, wallet, phone
   - Offer keywords: offer, free, donate, volunteer
   - Default: Help Needed

B) Urgency Detection:
   - High: urgent, emergency, critical, immediately
   - Low: whenever, no rush, flexible
   - Default: Medium

5.3 SECURITY IMPLEMENTATION
---------------------------

1. PASSWORD SECURITY
   - bcryptjs hashing with salt rounds
   - Password excluded from queries (select: false)
   - Minimum length validation

2. INPUT VALIDATION
   - Server-side validation on all endpoints
   - DOMPurify sanitization
   - Maximum length limits
   - Email format validation

3. RATE LIMITING
   - 5 posts per minute per IP
   - 3 reports per minute per IP
   - Prevents spam attacks

4. SESSION SECURITY
   - JWT-based sessions
   - HTTP-only cookies
   - CSRF protection
   - Secure transmission

5. DATA PROTECTION
   - Environment variables
   - MongoDB Atlas encryption
   - Sensitive field exclusion
